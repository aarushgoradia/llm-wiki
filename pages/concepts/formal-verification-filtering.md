## Definition

Using a formal verification tool (e.g., Cadence JasperGold) as a filter or reward signal in an LLM training pipeline, such that only formally-provable assertions are retained in training data or used as positive reinforcement. Distinct from using formal tools only to curate a static dataset.

## How It Works

In a filtering pipeline, candidate assertions generated by an LLM (or distilled from a larger model) are submitted to a formal checker. The checker attempts to prove each assertion against the RTL. Outcomes:

- **Proven:** assertion is correct and complete — retain as positive training signal
- **Counterexample found:** assertion is functionally wrong — discard or use as negative signal
- **Timeout/inconclusive:** assertion is either too complex to prove or the tool ran out of resources — typically discarded

This creates a verified subset of LLM outputs that is guaranteed correct by formal proof, not just by mutation testing or simulation sampling.

## Why It Matters

Mutation testing (VERT's validation method) works by inserting targeted bugs and checking that assertions catch them. It provides good coverage of the specific mutation operators used, but:

1. It cannot prove absence of bugs — only that the chosen mutants are caught
2. It does not validate assertions against the full state space of the design
3. Assertions that pass mutation testing may still be formally unprovable (vacuously true, over-constrained, or dependent on assumptions not present in the design)

Formal verification provides a stronger guarantee: if JasperGold proves an assertion, it holds for all reachable states under the given constraints. This is the standard of correctness in industrial verification flows.

## The VERT Gap

[[vert-dataset]] uses JasperGold to validate the *training data* during dataset construction, but JasperGold is not in the loop during model training or inference. The model is trained to imitate formally-validated examples, but receives no direct signal about formal provability. A model fine-tuned on VERT can produce assertions that:
- Are syntactically correct
- Pass mutation testing on the eval SoC
- Are not formally provable (e.g., due to missing clock constraints, vacuous antecedents, or incorrect assumption scope)

Veri2 addresses this by using JasperGold as a filter on GPT-5-distilled responses, keeping only formally-verified assertions for training.

## Practical Constraints

- **Scalability:** Formal verification is computationally expensive. Industrial designs can take hours to days per property. Filtering at training-data scale requires either restricting to small modules or using bounded model checking with tight timeouts.
- **Wrapper format:** JasperGold requires assertions in `assert property(propName);` form. VERT-style bare `property ... endproperty` blocks must be post-processed before formal elaboration — a non-trivial step at scale.
- **Async resets:** JasperGold assumes async reset signals remain inactive during execution. Assertions involving async resets require simulation (Xilinx Vivado) as a complement — pure formal filtering is incomplete without this fallback.

## Bidirectional Translation as Equivalence Proxy

[[2026-wu-codev-sva]] introduces a complementary technique: SVA→NL→SVA' round-trip with JasperGold equivalence checking as a proxy for NL-SVA semantic alignment. If the round-trip SVA' is formally equivalent to the original SVA, the intermediate NL description is likely faithful. This catches two failure modes that formal verification alone misses:

1. **Vacuous assertions:** `assert property (1'b1)` passes JasperGold against any RTL, but back-translates to a trivially obvious NL property, making the misalignment visible
2. **Operator precedence bugs:** subtle SVA syntax errors (e.g., `and` binding tighter than `|->`) produce non-equivalent re-translations, exposing the error

**Critical limitation:** this is *design-relative* equivalence, not semantic equivalence of the SVA formulas in isolation. JasperGold checks whether SVA ≡ SVA' on the specific RTL design c_i — i.e., for all traces of c_i, SVA holds iff SVA' holds. If c_i does not exercise a particular state, two assertions that diverge in that state will appear equivalent. Subtle NL-SVA misalignment can survive the filter when the test RTL doesn't cover the distinguishing behavior. The technique is a strong heuristic, not a formal semantic guarantee.

## Key Papers

- [[2025-menon-vert]] — formal verification used in dataset curation only, not in training loop; establishes the baseline that formal-filtered approaches improve upon
- [[2026-wu-codev-sva]] — most complete instantiation: JasperGold in Stage 1 (initial SVA curation) and Stage 2 (bidirectional equivalence checking); outputs proper `assert property()` wrappers

## Open Questions

- What fraction of VERT-style outputs fail formal verification on real SoCs, and what are the most common failure modes?
- Can bounded model checking with aggressive timeouts provide sufficient filtering at the scale needed for LLM training data?
- Would VERT + JasperGold filtering in the training pipeline close the gap with distillation-based approaches like Veri2?
- Does design-relative bidirectional equivalence provide sufficient semantic alignment guarantees, or do misaligned NL-SVA pairs survive when test RTL doesn't exercise distinguishing states?
