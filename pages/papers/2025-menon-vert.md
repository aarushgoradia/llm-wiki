## Summary

VERT is a 20,000-sample open-source dataset of (RTL snippet, SystemVerilog Assertion) pairs designed to fine-tune LLMs for hardware verification. Fine-tuning Llama 3.1 8B and DeepSeek Coder 6.7B on VERT produces models that beat GPT-4o by up to 24.14% on functional assertion correctness across four open-source RISC-V SoCs.

## Contributions

- First open-source dataset specifically targeting SVA generation, structured to address four documented GPT-4o failure modes
- Synthetic data generation pipeline: variables extracted from BOOM-core, rocket-chip, XiangShan + randomly generated names to prevent naming-convention overfitting
- Validation methodology combining mutation testing, Cadence JasperGold (formal), and Xilinx Vivado (simulation for async resets)
- Ablation study demonstrating that variable name cleanliness is the dominant factor in model performance — more important than model architecture
- Data leakage verification via 13-gram Jaccard similarity (overlap scores ~10⁻⁴)

## Method

**Dataset construction:** Variables are harvested from open-source HDL projects, deduplicated, and cleaned. Synthetic RTL snippets are generated by algorithmically combining variables with structural templates (if-else, case, combined). Assertions are derived deterministically from the conditional logic and verified before inclusion.

**Composition:** 20,000 samples — 52% if-else, 28% case, 20% combined; evenly split synchronous/asynchronous. The if-else emphasis directly targets GPT-4o's worst failure mode.

**Fine-tuning:** LoRA rank=256, alpha=256, max sequence length 4096 (needed for long condition chains), 3 epochs, batch size 64, lr=1×10⁻⁴, BF16. Targets Query/Key/Value/Output/Gate layers — 4–6% of parameters updated.

**Evaluation:** Four open-source SoCs — OpenTitan, CVA6, Pulpissimo, OpenPiton. Metrics: syntactic correctness (%), functional correctness (%), Complete Path Coverage (CPC). Functional validation: mutation testing + JasperGold + Vivado.

**Four GPT-4o failure modes VERT is designed to fix:**
1. *Clock cycle misinterpretation* — using `|=>` (non-overlapping) vs `|->` (overlapping) based on `always` block type
2. *if→else branch omission* — failing to negate preceding conditions when generating assertions for else branches
3. *Nested if-else* — resorting to ternary operators (invalid in SVA) for blocks nested beyond two levels
4. *Long conditions* — splitting a single multi-condition expression into multiple incorrect assertions

## Results

- DeepSeek Coder: base 19.91% → fine-tuned 96.25% functional correctness (avg across benchmarks)
- Llama 3.1: base 22.50% → fine-tuned 93.89% functional correctness
- Both models beat GPT-4o by up to 24.14% functional correctness and 20.69% syntactic correctness
- 100% CPC achieved on several benchmarks (OpenTitan/ADC CTRL, CVA6/Decode&Issue, Pulpissimo/APB, Pulpissimo/RISCV)
- Contamination (ifdef + module instantiations): negligible effect below 10 instances; ~3% degradation above 20 (worst-case, unrealistic in production)

**Ablation — variable name quality (Llama 3.1, avg syntactic correctness):**

| Data condition | Syntactic correct (%) |
|---|---|
| Base model | 34.65 |
| Syntactically incorrect vars | 32.88 |
| Duplicate vars | 77.66 |
| Inconsistent vars | 91.18 |
| Cleaned vars | 94.21 |

Variable cleanliness dominates. Training on bad names is worse than not fine-tuning at all.

## Limitations

**[First-hand, from building Veri2 on top of VERT]**

1. **Bare property declarations (no `assert property` wrapper).** VERT outputs assertions as bare `property ... endproperty` blocks, not as `assert property(propName);` statements. JasperGold requires the full assertion wrapper to elaborate and check a property. Any downstream formal verification pipeline must post-process VERT outputs to add wrappers before JasperGold can run — this is non-trivial for batch evaluation and introduces a gap between what VERT trains for and what formal tools consume.

2. **Snippet-level task framing, not module-level.** VERT trains on small synthetic RTL snippets (a single always block, a handful of signals). Real verification targets are full RTL modules with dozens of signals, multiple clock domains, and cross-module dependencies. The distribution shift between training snippets and full-module inference is uncharacterized.

3. **No formal verification step in the training pipeline.** VERT's validation uses mutation testing + Cadence JasperGold + Xilinx Vivado, but only to curate the *dataset* — JasperGold is not in the loop during training or as a reward signal. The model learns from syntactically and mutation-validated examples, but is never trained against formal provability. This means the model can produce assertions that pass syntax and mutation checks but are not formally provable — a distinction that matters in industrial formal verification flows. See [[formal-verification-filtering]].

4. **CPC is structural, not semantic.** 100% Complete Path Coverage means all automaton paths are exercised — it does not mean the right properties were written. A model can achieve 100% CPC while omitting security-critical or timing-critical behaviors entirely.

5. **Evaluation limited to open-source RISC-V SoCs.** All four evaluation platforms (OpenTitan, CVA6, Pulpissimo, OpenPiton) are open-source RISC-V. Generalization to proprietary architectures, non-RISC-V designs, or analog/mixed-signal blocks is unknown.

6. **GPT-4o baseline may be stale.** The paper compares against GPT-4o but not reasoning-class models (o1, o3, GPT-5). Reasoning models may handle the structured logical deduction required for nested conditions significantly better.

## Open Questions

- Does VERT's snippet-level training distribution transfer to full RTL modules, and how large is the performance gap?
- Would adding JasperGold as a filter or reward signal in the training pipeline (rather than only in dataset curation) close the functional correctness gap with distillation-based approaches?
- How much of VERT's gain over GPT-4o survives against reasoning-class models on the same benchmarks?
- Is there a semantic completeness metric that would better capture assertion coverage than CPC?
